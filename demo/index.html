Database Credentials

Defaults point at this repo's local docker-compose Postgres (seeded with sample rows by docker/postgres/init.sql). Both the "SQL Functions" and "SQL Interception" Workflows below run against this connection — SQL Functions via psql inside the running postgres container (see demo/server.py); SQL Interception via sql-interception-proxy's own fixed connection (env vars, not this form).

SQL Interception Rule Set

Columns sql-interception-proxy masks transparently in any SELECT's result, matched by column label (case-insensitive) — see the Workflows tab's "SQL Interception" panel.

ColumnSchemeDelphix Algorithm
full_nameFULL-NAME-MASKFullNameMaskingAlgorithm
emailEMAIL-MASKEmailMaskingAlgorithm
credit_cardCREDIT-CARD-MASKCreditCardMaskingAlgorithm

SQL Functions Rule Set (opt-in)

Every SQL UDF the "SQL Functions" workflow below can call: the scheme ("algorithm framework" in OneWayMaskingSchemeRegistry/ TokenizationSchemeRegistry) it's permanently fixed to, and the Delphix Continuous Compliance engine algorithm class that runs the same scheme (see FixedSchemeMaskingAlgorithm). For each demo column, the selector in the last column rewrites the Masked Data SQL workflow.

SQL UDFSchemeDelphix AlgorithmDemo column
mask_fullname() FULL-NAME-MASK FullNameMaskingAlgorithm
mask_email() EMAIL-MASK EmailMaskingAlgorithm
mask_creditcard() CREDIT-CARD-MASK CreditCardMaskingAlgorithm
mask_firstname() FIRST-NAME-LOOKUP FirstNameMaskingAlgorithm
mask_lastname() LAST-NAME-LOOKUP LastNameMaskingAlgorithm
mask_dateshift() DATE-SHIFT DateShiftMaskingAlgorithm
mask_segmentmapping() SEGMENT-MAPPING SegmentMappingMaskingAlgorithm
tokenize() / reidentify() AES-CBC-CTS (default) TokenizationAlgorithm any column — reversible; also selectable per row above

SQL Functions: Sensitive Data (raw)

Opt-in — masking here only happens when the query explicitly calls tokenize()/mask_*(), unlike the SQL Interception workflow above.

SQL Functions: Masked Data

One function per data shape -- always available, nothing to configure: mask_fullname(), mask_creditcard() (keeps separators), mask_email() (keeps the domain). tokenize() (reversible) works on any column too — see the SQL Functions Rule Set on the Rule Sets tab.

SQL Interception

          
Runs any SELECT below — it never calls
            tokenize()/mask_*() — and sql-interception-proxy
            masks known-sensitive columns (full_name/email/
            credit_card) in the result transparently. Columns highlighted below were
            masked. Known limitation: an aliased sensitive column (e.g.
            SELECT email AS contact) isn't recognized by column label alone and
            passes through unmasked. This is a best-effort SELECT-only guard, not a
            security boundary.

Sync Keys

      
Privileged action, cannot be reversed. Use with caution.

      
Syncs the attached engine's configured data encryption key into this
        deployment's .env — the reverse direction of
        scripts/sync-engine-config.sh, which pushes a local key to the engine instead.
        Only works against the local docker-compose stack (rebuilds tokenization-api
        afterward, same as "Apply Configuration").
