Algorithm Config JSON (for the Delphix engine)

It's independent of this demo's KEY_SOURCE/KMS envelope setup on the Connectors tab: the engine plugin only takes the key inline (or via keyFile), never KMS-wrapped. SQL Functions Rule Set (opt-in)

Every SQL UDF the "SQL Functions" job below can call: the scheme ("algorithm framework" in OneWayMaskingSchemeRegistry/ TokenizationSchemeRegistry) it's permanently fixed to, and the Delphix Continuous Compliance engine algorithm class that runs the same scheme (see FixedSchemeMaskingAlgorithm). For each demo column, the selector in the last column rewrites the Masked Data SQL job.

SQL Interception

Distinct from the "SQL Functions" jobs below: run any SELECT below — it never calls tokenize()/mask_*() — and sql-interception-proxy masks known-sensitive columns (full_name/email/credit_card) in the result transparently. Columns highlighted below were masked. Known limitation: an aliased sensitive column (e.g. SELECT email AS contact) isn't recognized by column label alone and passes through unmasked. This is a best-effort SELECT-only guard, not a security boundary.

SQL Functions: Sensitive Data (raw)

Opt-in — masking here only happens when the query explicitly calls tokenize()/mask_*(), unlike the SQL Interception job above.

SQL Functions: Masked Data

One function per data shape -- always available, nothing to configure: mask_fullname(), mask_creditcard() (keeps separators), mask_email() (keeps the domain). tokenize() (reversible) works on any column too — see the SQL Functions Rule Set on the Rule Sets tab.

Sync Key from Delphix Engine

Authenticates against a live Delphix Masking Engine and syncs its configured data encryption key into this deployment's .env — the reverse direction of scripts/sync-engine-config.sh, which pushes a local key to the engine instead. Only works against the local docker-compose stack (rebuilds tokenization-api afterward, same as "Apply Configuration").
